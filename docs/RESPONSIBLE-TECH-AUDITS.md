# Responsible-Tech Audits — Olive's Bark Logger

**Last verified: 2026-07-05 · Reviewer: Chelsea Kelly-Reif · Recheck cadence: per release + per STANDARDS remediation pass.**

Instantiates `/STANDARDS/RESPONSIBLE-TECH-FRAMEWORK.md`. A device that listens in a home has obvious privacy and recording-law stakes; the entire design answer is "measure levels, never audio." Checklists run in CI from M0; artifacts are committed and regenerated by `make verify`.

---

## A. Ethics & responsibility
- **Worst misuse:** recording the household or neighbors, or using cherry-picked numbers to manufacture a case against a neighbor.
- **Mitigations:** audio is never captured, stored, or transmitted (tested); the report states methodology and limitations and makes no source-attribution claims; the framing is informational, not adversarial.
- **"Works as intended" harm:** bare numbers read as more certain than they are. → Mandatory limitations section.
- **Non-goals:** not surveillance; not a recorder. **Auto-gated:** no-audio + no-egress + report-content tests. **Review-gated:** ethics review (honest use).

## B. Bias & fairness
- **Segments:** time-of-day and threshold sensitivity.
- **Risks:** miscalibration or a poorly placed device systematically over- or under-counting events, skewing the picture.
- **Tests:** detection validated against a labeled session; calibration and placement documented.
- **Commitment:** report measurement conditions; never present tuned numbers as objective truth. **Auto-gated:** eval test presence. **Review-gated:** methodology review.

## C. Privacy & data-protection
- **Data inventory:** event metadata only (timestamps, durations, levels). **No audio anywhere.**
- **Handling:** in-memory frame processing with immediate discard; local SQLite; no network.
- **Commitment:** no recording, no transmission, minimal retention. **Auto-gated:** no-audio + no-egress tests. **Review-gated:** privacy review.

## D. Transparency & explainability
- **In the report:** how levels are measured, the calibration offset, thresholds used, and explicit limitations (relative dBFS, no source attribution).
- **Commitment:** the report explains itself and states what it cannot prove. **Auto-gated:** report-content test (methodology + limitations present). **Review-gated:** framing review.

## E. Accessibility (WCAG 2.2 AA)
- **Report/dashboard:** keyboard-complete; every chart has a data-table equivalent; levels/severity not color-only.
- **Commitment:** zero `axe` violations plus a committed manual walkthrough. **Auto-gated:** axe + keyboard + chart-table tests. **Review-gated:** SR walkthrough sign-off.

## F. Security
- **Threat model:** tampering with the event log, dependency risk; minimal attack surface since there's no network or audio.
- **Controls:** local-only operation, input validation, SAST + deps + secret scans merge-blocking.
- **Residual-risk register** committed in `docs/audits/`. **Auto-gated:** scanners. **Review-gated:** threat-model sign-off.
- **ASVS level:** **L2** (per `SECURITY-AND-SUPPLY-CHAIN-STANDARD.md` §1 — this repo is named as an L2 example: it handles PII-adjacent household-noise data, even though that data is levels/timestamps only, never audio).
- **Container scanning:** enabled — `aquasecurity/trivy-action` runs against the built image in `.github/workflows/ci.yml`'s verify job (`CRITICAL,HIGH`, blocking). Added 2026-07-05; not yet re-verified against a live CI run in this sandbox (no local Docker daemon available at remediation time — see `docs/GAP-LEDGER.md` note under CI/CD if this needs a first-run check).
- **SBOM + signing:** not yet — no release pipeline exists to attach an SBOM or signature to (tracked: `docs/GAP-LEDGER.md#gap-rel-1--release--versioning-the-releasesupply-chain-pipeline-is-still-absent`). Not "N/A": REL §1 explicitly names this repo release-producing, so this is an open gap, not an opt-out.
- **Secret management:** no secrets exist in this repo by design (local-only, no API keys, no cloud creds); CI uses only the ephemeral, read-only, auto-issued `GITHUB_TOKEN` (`permissions: contents: read` at the workflow level, `.github/workflows/ci.yml`). `gitleaks` runs merge-blocking on every push/PR as the verification layer.
- **VEX (unfixable dependency CVEs):** ten dev-toolchain-only CVEs are waived in `Makefile`'s `security` target (`PIP_AUDIT_WAIVERS`), dated 2026-07-05. All ten are in `pip-audit`'s own transitive dependencies (filelock, msgpack, pip, pytest, requests, urllib3) — never shipped in the runtime, which has zero dependencies (`pyproject.toml`) — and every available fix requires a Python ≥3.10 host, which this repo's documented ≥3.9 dev-venv floor (`ADR-0002`) cannot yet install. Re-audit when the Python-floor decision changes or a <3.10-compatible fix ships. This supersedes an earlier, now-stale "VEX: none needed" draft from the 2026-07-05 remediation plan — that plan was written before hardening `make security` from a soft-skip to a hard-fail surfaced these ten findings for the first time.
